02 · Why Kotlin?

Kotlin is Java — but Way Better

🎯
Null Safety Built-In
Kotlin forces you to handle nulls at compile time. No more random NullPointerException crashes at runtime that you only discover in production.
✨
Data Classes — Zero Boilerplate
One line gives you equals(), hashCode(), copy(), toString(). In Java that's 60+ lines of generated code nobody wants to write.
⚡
Coroutines for Async Code
Ktor is built on coroutines. Handle thousands of HTTP requests concurrently — written as simple, readable sequential code.
🔗
100% Java Interop
I use jBCrypt (a Java library) directly from Kotlin — no wrappers needed. All the Java ecosystem, none of the Java verbosity.
model/Event.kt — a data class in practice
// ONE LINE = full model with equals, copy, toString data class Event( val id: Long, val title: String, val description: String, val organizerId: Long? = null, // ? = nullable val maxParticipants: Int? = null ) // Kotlin lets you do this: val updated = event.copy(title = "New Title") // event == otherEvent checks by VALUE, not reference
Bottom line

Kotlin lets me focus on business logic, not ceremony. Less code = fewer bugs = faster development. That's why Ktor (Kotlin server framework) is my choice.

05 · Design Patterns

4 Patterns — Each Solving a Real Problem

Ports & Adapters
aka Hexagonal

Problem: if EventService called JDBC directly, you'd need a real database just to run a test. Fix: services depend on the EventDao interface. In tests a fake is injected, in prod the real one is used. Zero code change.

Singleton

Problem: creating a new RabbitMQ connection for every single message would exhaust resources fast. Fix: RabbitMQConnectionFactory reads env vars once and holds a single shared config for the app's entire lifetime.

Strategy

Problem: I need real emails in prod, console logs in dev, and captured messages in tests — without changing any service code. Fix: NotificationPublisher is an interface. Swap RabbitMQPublisher / LogPublisher / FakePublisher at wiring time.

Template Method

Problem: two notification consumers share the same connect-listen loop — copy-pasting it is a maintenance nightmare. Fix: abstract NotificationConsumer defines the loop once. Subclasses just override handleMessage().

All four patterns serve the same goal: write the same thing once, make it easy to test, make it easy to change.

06 · Code Spotlight — Ports & Adapters

The Interface Trick That Makes Tests Instant

db/dao/EventDao.kt — The PORT (just an interface)
// No SQL here. Just a contract. interface EventDao { fun findAll(): List<Event> fun findById(id: Long): Event? fun create(event: Event): Event fun update(id: Long, event: Event): Event? fun delete(id: Long): Boolean }
service/EventService.kt — uses the PORT
class EventService( // depends on the interface, NOT the DB private val eventDao: EventDao ) { fun createEvent(req: EventRequest): EventResponse { validate(req) // business rules first val created = eventDao.create(req.toModel()) return created.toResponse() } }
fake/FakeEventDao.kt — used in TESTS
class FakeEventDao : EventDao { // just a HashMap — zero DB needed! private val store = mutableMapOf<Long, Event>() private var nextId = 1L override fun create(event: Event): Event { val saved = event.copy(id = nextId++) store[saved.id] = saved return saved } override fun findAll() = store.values.toList() override fun findById(id) = store[id] override fun delete(id) = store.remove(id) != null }
💡 Why this is powerful

EventService doesn't know if it's talking to PostgreSQL or a HashMap. Tests run in milliseconds with no Docker, no DB setup, no cleanup.

07 · Code Spotlight — Strategy & Template Method

One Interface, Three Behaviours

messaging/NotificationPublisher.kt — the Strategy
// The interface is the "contract" interface NotificationPublisher { suspend fun publish(msg: NotificationMessage) } // Production: sends to RabbitMQ class RabbitMQNotificationPublisher(...) : NotificationPublisher { override suspend fun publish(msg) { channel.basicPublish(Json.encode(msg), routingKey) } } // Dev fallback: just logs to console class LogNotificationPublisher : NotificationPublisher { override suspend fun publish(msg) { logger.info("[NOTIFICATION] {}", msg.eventType) } }
notification-service/NotificationConsumer.kt — Template Method
// Abstract class = fixed skeleton, variable step abstract class NotificationConsumer(...) { // subclasses MUST fill this in protected abstract fun handleMessage(msg: NotificationMessage) suspend fun start() { // this part is FIXED — connect, declare, loop channel.basicConsume(queue) { delivery -> val msg = Json.decode(delivery.body) handleMessage(msg) // calls subclass } awaitCancellation() // keep running forever } } // Subclass only writes what's different: class UserNotificationConsumer(...) : NotificationConsumer(...) { override fun handleMessage(msg) { // handle USER_REGISTERED event type } }
Think of it like a recipe 🥣

Template Method = the recipe steps are fixed (boil water, steep, pour). Strategy = you choose which tea. Both say: write the varying part only once.

09 · Auth & Security

JWT in a HttpOnly Cookie — Simple & XSS-Proof

security/JwtManager.kt
class JwtManager( private val secret: String = System.getenv("JWT_SECRET") ) { fun generateToken(userId, email, role): String = JWT.create() .withSubject(userId.toString()) .withClaim("email", email) .withClaim("role", role.name) .withExpiresAt(Date(now + 24h)) .sign(Algorithm.HMAC256(secret)) fun verifyToken(token: String): JwtClaims? = try { val decoded = verifier.verify(token) JwtClaims(decoded.subject.toLong(), ...) } catch (e: JWTVerificationException) { null } }
security/PasswordUtil.kt
object PasswordUtil { fun hashPassword(p: String) = BCrypt.hashpw(p, BCrypt.gensalt(12)) // cost=12 = slow by design. Hard to brute-force. fun verifyPassword(plain, hash) = BCrypt.checkpw(plain, hash) // plain-text password NEVER stored }
Why HttpOnly cookie?

If the JWT is in localStorage, malicious JavaScript on the page can steal it. HttpOnly means document.cookie returns nothing for it — XSS-proof by default.

Password Rules (enforced in AuthService)

Hashed with bcrypt cost 12 — irreversible even if the DB is leaked.

Dual-mode auth: browser uses the cookie automatically; API tests use Authorization: Bearer <token> header. Same JWT verifier for both.

Outbox pattern — scheduled reminders stored here, polled async.

🔄 Re-registration trick

UNIQUE constraint on (student_id, event_id) prevents duplicates. When re-registering after cancel, I UPDATE the row back to REGISTERED instead of inserting — history preserved, constraint respected.

Flyway runs V1.sql → V7.sql automatically at startup. Schema changes version-controlled like code.

14 · Code Spotlight — Registration Flow

What Happens When a Student Registers for an Event

service/RegistrationService.kt (simplified)
fun registerStudent(studentId: Long, eventId: Long) { // 1. Does the event exist? val event = eventDao.findById(eventId) ?: throw IllegalArgumentException("Event not found") // 2. Already registered? Throw, not silent. val existing = regDao.findByStudentAndEvent(studentId, eventId) if (existing?.status == "REGISTERED") throw IllegalArgumentException("Already registered") // 3. Capacity check event.maxParticipants?.let { max -> if (regDao.countActive(eventId) >= max) throw IllegalArgumentException("Event is full") } // 4. Re-register or create? return if (existing != null) { regDao.reactivate(existing.id).toResponse() } else { regDao.create(Registration(studentId, eventId)).toResponse() } }
